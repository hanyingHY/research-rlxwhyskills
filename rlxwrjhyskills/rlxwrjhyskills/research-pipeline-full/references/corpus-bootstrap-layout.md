# Corpus Bootstrap Layout

## Objective

Provide a generic starting structure for research projects that begin as an unstructured pile of files rather than a clean program workspace.

## When to use it

Bootstrap first when one or more are true:

1. the workspace is mostly loose documents, papers, exports, screenshots, or notes
2. there is no stable intake area or no visible distinction between raw material and judged material
3. multiple workers will need a shared structure before they can safely divide work
4. prior research content exists, but it is not yet separated into trusted, untrusted, and undecided layers

## Minimum bootstrap zones

Create or identify these functional zones before deeper research proceeds:

1. raw intake
2. source index
3. retained evidence
4. conflict or adjudication log
5. validation or action mapping
6. reports and user-facing decisions
7. quarantine or reject area for weak, duplicate, or suspicious material

## Default bootstrap policy

1. keep the original raw material intact
2. avoid renaming everything up front just for tidiness
3. assign stable locations for structured outputs before workers start writing
4. create a visible intake-to-judgment path so later reviewers can tell what was imported, what was retained, and what was rejected
5. do not treat pre-existing summaries as trusted until they survive the existing-content audit

## Bootstrap output contract

After bootstrapping, the operator should be able to answer:

1. where new material lands
2. where deduplicated source records live
3. where retained claims live
4. where conflicts are logged
5. where the next user decision is recorded
6. where weak or suspicious material is quarantined

## Rule

Bootstrapping is not a substitute for evidence grading.

It only creates a safe operating structure so the later audit, discovery, synthesis, and validation phases can be run without hidden file chaos.
